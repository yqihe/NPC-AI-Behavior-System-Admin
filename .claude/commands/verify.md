# /verify — 代码验证

验证代码是否正确。必须跑真实命令，最终判定 PASS 或 FAIL，不允许模糊结论。

## Usage
```
/verify <feature-name>
/verify <feature-name> --task=T1    # 只验证某个任务的改动
```

---

## 核心原则

**不是确认代码能工作，而是想方设法把它搞崩。**

如果你发现自己在想"这个应该不会出问题"——停下来，做相反的事，去试试能不能搞崩它。

---

## 执行流程

1. 加载 spec 的 requirements.md，提取验收标准（R1, R2, ...）
2. **在跑之前**，列出每条验收标准对应的具体验证方法和预期结果
3. 执行验证（必须跑真实命令）
4. 对每条标准判定 PASS / FAIL
5. 输出验证报告

---

## 验证手段（按优先级）

> **只执行与本次任务相关的手段，无关的跳过并在报告中注明原因**（例如纯前端任务不跑 go build，纯 store 层改动不开浏览器）。

### 后端
1. **`cd backend && go build ./...`** — 编译通过
2. **`cd backend && go test ./...`** — 单元测试全部通过
3. **`cd backend && go test -race ./...`** — 并发代码无竞态
4. **curl 测试 API** — 手动验证端点响应格式、状态码、错误处理
5. **mongosh 验证数据** — 检查写入 MongoDB 的文档格式是否符合 `{name, config}`

### 前端
6. **`cd frontend && npm run build`** — 构建通过，无 TypeScript/ESLint 错误
7. **浏览器手动验证** — 页面渲染、表单交互、提交流程
8. **浏览器控制台** — 无 JS 错误、无 Vue warning

### 综合
9. **`docker compose up --build`** — 容器编排正常启动
10. **手动构造边界输入** — 空值、重复名称、超大 body、非法引用

---

## 禁止

- **不准只看代码就说通过**——必须跑真实命令，`go test` 输出是唯一证据
- **不准模糊结论**——每条验收标准必须判 PASS 或 FAIL，不存在"基本通过"、"应该没问题"
- **不准给自己找借口**——"这个测试不稳定所以跳过"不可接受
- **不准修改测试来迁就代码**——除非先确认服务端行为符合设计预期
- **不准跳过竞态检测**——涉及并发的代码必须跑 `-race`

## 测试环境规范

运行 `tests/integration_test.sh` 前，脚本自动执行环境重置。手动跑 curl 测试时也必须遵守（详见 `docs/development/admin/dev-rules.md` "测试环境重置（Phase 0）"章节）：

1. **Redis FLUSHALL** — 清除上次运行的缓存残留
2. **DROP + CREATE 业务表** — 重置 AUTO_INCREMENT
3. **保留字典表** — 有种子数据的表不重建
4. **先 seed 后重启后端** — DictCache 启动时加载
5. **所有 jq 提取必须 `| tr -d '\r'`** — Windows CRLF 问题

## 攻击性测试思维

跑完基础测试后，主动尝试搞崩它：

### 后端
- **空值/缺字段**：提交空 name、空 config、缺少必填字段
- **重复创建**：同 name 创建两次，验证返回 409 而非 500
- **非法引用**：NPC 类型引用不存在的 FSM/BT，验证返回 422
- **删除被引用项**：删除正被 NPC 引用的 FSM，验证拒绝删除
- **超大 body**：提交 > 1MB 的请求体，验证被拦截
- **注入攻击**：name 字段传 `{"$gt": ""}`，验证不会绕过查询
- **并发写入**：两个请求同时创建同 name 配置

### 前端
- **快速双击提交**：按钮是否有防重复
- **空列表状态**：无数据时页面不 crash
- **表单残留**：关闭新建对话框再打开，数据是否清空
- **网络错误**：后端宕机时前端是否有友好提示

## 失败处理

断言 FAIL 时，**先定位是代码 bug 还是测试问题**：

1. 查服务端日志/输出，确认实际行为是什么
2. 对照 design.md，判断实际行为是否符合设计
3. **实际行为不符合设计** → 代码 bug → 建议 `/debug`
4. **实际行为符合设计但测试预期错误** → 修改测试
5. **实际行为符合设计但设计本身有问题** → 回到 spec 修改设计，需要用户审批

## 验证报告模板

```
## 验证报告

**Spec**：<feature-name>
**范围**：全部 / T[N]
**当前 commit**：[git log --oneline -1]

### 验收标准检查
| 标准 | 验证方法 | 预期 | 实际 | 判定 |
|------|----------|------|------|------|
| R1: xxx | go test -run TestXxx | PASS | PASS | PASS |
| R2: xxx | e2e 场景 A | 状态变为 Flee | 状态变为 Flee | PASS |

### 攻击性测试
| 测试项 | 结果 |
|--------|------|
| nil 输入 | PASS - 返回错误，不 panic |
| 并发读写 | PASS - race detector 无报告 |

### 结论
**PASS** / **FAIL**（X 条失败）

### 下一步
- PASS → 提交 commit，**停下等用户审批后再决定是否进入下一个 task**（与 `/spec-execute` 步骤 9 一致）
- FAIL → 跑 `/debug <symptom>`
```
