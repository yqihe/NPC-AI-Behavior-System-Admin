# /debug — 故障排查

接受一个症状，定位根因，修复。

## Usage
```
/debug <症状描述>
/debug "NPC 收到事件后没有切换状态"
/debug "go test -race 报竞态"
/debug "e2e 超时，没收到 npc.state_change"
```

---

## 排查原则

**不准猜，用证据定位。**

每一步排查必须有命令输出或日志作为证据。"我觉得可能是"不算定位。

---

## 排查流程

### Step 1：分类

根据症状判断故障域：

| 域 | 典型症状 |
|----|----------|
| 后端编译/启动 | go build 失败、panic at startup、import cycle |
| MongoDB/Redis | 连接失败、超时、查询无结果、缓存脏数据 |
| API 行为 | 状态码错误、响应格式异常、数据没写入、缓存没失效 |
| 数据校验 | 校验该拦没拦、引用检查遗漏、合法输入被拒 |
| 前端构建 | npm run build 失败、TypeScript 类型错误、依赖缺失 |
| 前端运行时 | 页面白屏、控制台报错、Vue warning、响应式不更新 |
| 前端交互 | 表单校验不触发、弹窗数据残留、请求竞态 |
| Docker | 容器启动失败、端口冲突、服务间网络不通 |

### Step 2：假设 → 验证

针对故障域，提出最可能的假设，用真实命令验证：

- **一次只验证一个假设**
- 假设被否定 → 提下一个假设
- 假设被确认 → 进入修复

### Step 3：修复

- **一次只改一个点**——不准同时改多个可疑位置
- 改完后检查对应技术领域的陷阱文档（`docs/development/` 下按技术拆分），确认修复没有引入新问题

### Step 4：验证修复

- 修完必须回到 `/verify` 确认问题解决且没有引入新问题
- 不准改了就走

---

## 禁止

- **不准猜测性修复**——没有定位到根因就改代码
- **不准一次改多处**——改一个点，验证，再改下一个
- **不准只改代码不验证**——每次修改后必须跑测试
- **不准忽略 race detector**——竞态问题必须修，不准用"偶发性问题"搪塞
- **不准删测试来消除失败**

## 排查报告模板

```
## 故障排查报告

**症状**：[原始描述]
**故障域**：[分类]

**排查过程**：
1. 假设：[假设内容]
   验证：[执行的命令]
   结果：[确认/否定]

2. 假设：[假设内容]
   验证：[执行的命令]
   结果：[确认 → 定位到根因]

**根因**：[一句话]
**修复**：[改了什么]
**验证**：[跑了什么测试，结果]

→ 建议跑 `/verify <feature-name>` 完整验证
```

## 经验沉淀

排查过程中发现的新坑，按技术领域追加到对应文档（参考 `docs/development/dev-rules.md` 经验沉淀指引表格）。
