# 禁止事项（红线）

## 禁止暴露技术细节给策划

- **禁止**在 UI 中展示原始 BB Key 名称（如 `threat_level`）。必须用友好的中文标签（如"威胁等级"）
- **禁止**让策划手写 JSON。所有配置通过表单组件输入，后端负责转换为正确的 JSON 格式
- **禁止**让策划看到报错堆栈或 Go error 信息。错误提示必须是人类可读的中文描述

## 禁止破坏游戏服务端数据格式

- **禁止**修改 MongoDB collection 结构（`{name, config}` 格式由游戏服务端定义）
- **禁止**在 config 字段中添加游戏服务端不认识的额外字段（如 UI 状态、排序信息）。如需存储运营平台私有数据，使用独立的 collection
- **禁止**删除正在被其他配置引用的条目（如删除一个 FSM 配置时，需检查是否有 NPC 类型引用它）

## 禁止安全隐患

来源：游戏服务端代码审查发现的教训。

- **禁止**将用户输入直接拼接为数据库查询。使用参数化查询防止注入
- **禁止**信任前端校验。所有校验必须在后端重复执行
- **禁止**在 API 响应中暴露 MongoDB ObjectId 或内部错误信息

## 禁止缓存与数据库不一致

- **禁止**写操作（CREATE/UPDATE/DELETE）成功后不清除对应 Redis 缓存。必须先写 DB，成功后立即删缓存（Cache-Aside 模式）
- **禁止**缓存 DELETE 失败时静默吞掉错误。删缓存失败必须记日志告警（降级为不缓存，不能返脏数据）
- **禁止**在缓存中存储无 TTL 的数据。所有缓存 key 必须设置过期时间作为兜底，防止极端情况下的永久脏缓存

## 禁止引用完整性破坏

- **禁止**删除正在被引用的配置（已有规则，此处强化细节）：
  - 删除 FSM 配置前，必须检查 `npc_types` 中是否有 `fsm_ref` 指向它
  - 删除 BT 配置前，必须检查 `npc_types` 中是否有 `bt_refs` 值指向它
  - 删除事件类型前，必须检查 FSM 条件中是否引用了该事件名
- **禁止**创建 NPC 类型时引用不存在的 FSM 或 BT 配置。保存时必须校验引用目标存在

## 禁止 MongoDB 操作符注入

- **禁止**将用户输入的 JSON key 直接用于 MongoDB 查询或文档字段。所有以 `$` 开头的 key 必须拒绝，防止操作符注入（如 `{"$gt": ""}` 绕过条件）
- **禁止**用 `bson.M` 拼接用户输入构造查询。name 字段查询必须用精确匹配 `bson.M{"name": name}`，不接受正则或操作符

## 禁止无超时的外部调用

- **禁止**使用 `context.Background()` 或 `context.TODO()` 直接调用 MongoDB / Redis。所有外部 IO 操作必须使用带超时的 context（`context.WithTimeout`），防止网络异常导致 handler 永久挂起
- **禁止** HTTP handler 内不设请求体大小限制。必须用 `http.MaxBytesReader` 限制 body 上限（1MB）

## 禁止与游戏服务端 Schema 不一致

来源：首次联调反馈，发现 5 处不一致，其中 inverter 节点分类错误导致产出的配置无法加载。

- **禁止**校验用结构体的字段类型与游戏服务端不一致。例如游戏服务端 `DefaultSeverity` 是 `float64`，运营平台必须也用 `float64`（不能用 `int`）。新增校验字段时必须查游戏服务端源码确认类型
- **禁止**将装饰节点（如 `inverter`）归类为复合节点。游戏服务端 `inverter` 使用 `child`（单个 `*TreeConfig`），不是 `children`（`[]TreeConfig`）。错误分类会导致产出的 JSON 格式游戏服务端无法解析
- **禁止**放行游戏服务端不支持的枚举值。FSM 条件的 `op` 操作符必须在游戏服务端支持的列表内（`==`、`!=`、`>`、`>=`、`<`、`<=`、`in`），`parallel` 的 `policy` 必须是 `require_all` 或 `require_one`，`stub_action` 的 `result` 必须是 `success` / `failure` / `running`。无效枚举在游戏服务端静默降级，不报错但行为不符预期，极难排查
- **禁止**写入未注册的 Blackboard Key。`set_bb_value` / `check_bb_float` / `check_bb_string` 节点的 `key` 必须在游戏服务端 `blackboard/keys.go` 的注册表白名单内。未注册的 key 写入 MongoDB 后，游戏服务端加载行为树时会直接 **panic**——这是生产事故级别的问题。前端必须使用下拉选择器（不允许手动输入），后端 validator 必须白名单校验

## 禁止过度设计

- **禁止**实现用户认证/权限系统。毕设阶段所有用户等权
- **禁止**实现配置版本控制/回滚。JSON 文件已有 Git 版本控制
- **禁止**实现实时协作编辑。单人编辑足够
- **禁止**实现复杂的工作流审批。保存即生效（到 MongoDB）
