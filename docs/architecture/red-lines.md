# 禁止事项（红线）

## 禁止暴露技术细节给策划

- **禁止**在 UI 中展示原始 BB Key 名称（如 `threat_level`）。必须用友好的中文标签（如"威胁等级"）
- **禁止**让策划手写 JSON。所有配置通过表单组件输入，后端负责转换为正确的 JSON 格式
- **禁止**让策划看到报错堆栈或 Go error 信息。错误提示必须是人类可读的中文描述

## 禁止破坏游戏服务端数据格式

- **禁止**修改 MongoDB collection 结构（`{name, config}` 格式由游戏服务端定义）
- **禁止**在 config 字段中添加游戏服务端不认识的额外字段（如 UI 状态、排序信息）。如需存储运营平台私有数据，使用独立的 collection
- **禁止**删除正在被其他配置引用的条目（如删除一个 FSM 配置时，需检查是否有 NPC 类型引用它）

## 禁止安全隐患

来源：游戏服务端代码审查发现的教训。

- **禁止**将用户输入直接拼接为数据库查询。使用参数化查询防止注入
- **禁止**信任前端校验。所有校验必须在后端重复执行
- **禁止**在 API 响应中暴露 MongoDB ObjectId 或内部错误信息

## 禁止缓存与数据库不一致

- **禁止**写操作（CREATE/UPDATE/DELETE）成功后不清除对应 Redis 缓存。必须先写 DB，成功后立即删缓存（Cache-Aside 模式）
- **禁止**缓存 DELETE 失败时静默吞掉错误。删缓存失败必须记日志告警（降级为不缓存，不能返脏数据）
- **禁止**在缓存中存储无 TTL 的数据。所有缓存 key 必须设置过期时间作为兜底，防止极端情况下的永久脏缓存

## 禁止引用完整性破坏

- **禁止**删除正在被引用的配置（已有规则，此处强化细节）：
  - 删除 FSM 配置前，必须检查 `npc_types` 中是否有 `fsm_ref` 指向它
  - 删除 BT 配置前，必须检查 `npc_types` 中是否有 `bt_refs` 值指向它
  - 删除事件类型前，必须检查 FSM 条件中是否引用了该事件名
- **禁止**创建 NPC 类型时引用不存在的 FSM 或 BT 配置。保存时必须校验引用目标存在

## 禁止 MongoDB 操作符注入

- **禁止**将用户输入的 JSON key 直接用于 MongoDB 查询或文档字段。所有以 `$` 开头的 key 必须拒绝，防止操作符注入（如 `{"$gt": ""}` 绕过条件）
- **禁止**用 `bson.M` 拼接用户输入构造查询。name 字段查询必须用精确匹配 `bson.M{"name": name}`，不接受正则或操作符

## 禁止无超时的外部调用

- **禁止**使用 `context.Background()` 或 `context.TODO()` 直接调用 MongoDB / Redis。所有外部 IO 操作必须使用带超时的 context（`context.WithTimeout`），防止网络异常导致 handler 永久挂起
- **禁止** HTTP handler 内不设请求体大小限制。必须用 `http.MaxBytesReader` 限制 body 上限（1MB）

## 禁止与游戏服务端 Schema 不一致

来源：首次联调反馈，发现 5 处不一致，其中 inverter 节点分类错误导致产出的配置无法加载。

- **禁止**校验用结构体的字段类型与游戏服务端不一致。例如游戏服务端 `DefaultSeverity` 是 `float64`，运营平台必须也用 `float64`（不能用 `int`）。新增校验字段时必须查游戏服务端源码确认类型
- **禁止**将装饰节点（如 `inverter`）归类为复合节点。游戏服务端 `inverter` 使用 `child`（单个 `*TreeConfig`），不是 `children`（`[]TreeConfig`）。错误分类会导致产出的 JSON 格式游戏服务端无法解析
- **禁止**放行游戏服务端不支持的枚举值。FSM 条件的 `op` 操作符必须在游戏服务端支持的列表内（`==`、`!=`、`>`、`>=`、`<`、`<=`、`in`），`parallel` 的 `policy` 必须是 `require_all` 或 `require_one`，`stub_action` 的 `result` 必须是 `success` / `failure` / `running`。无效枚举在游戏服务端静默降级，不报错但行为不符预期，极难排查
- **禁止**写入未注册的 Blackboard Key。`set_bb_value` / `check_bb_float` / `check_bb_string` 节点的 `key` 必须在游戏服务端 `blackboard/keys.go` 的注册表白名单内。未注册的 key 写入 MongoDB 后，游戏服务端加载行为树时会直接 **panic**——这是生产事故级别的问题。前端必须使用下拉选择器（不允许手动输入），后端 validator 必须白名单校验

## 禁止绕过后端 API 直接操作 MongoDB

来源：导入种子数据时直接用 mongosh 插入，Redis 缓存未失效，前端读到空数据。

- **禁止**开发/调试/导入数据时直接用 mongosh 或其他客户端写入 MongoDB。所有数据变更（包括种子数据导入）必须通过后端 REST API（`curl http://localhost:9821/api/v1/...`），保证 Redis 缓存同步失效
- **禁止**在脚本中直接连 MongoDB 批量写入。如需批量导入，编写脚本逐条调用后端 API

## 禁止并发资源关闭不安全

来源：审查 Mailbox 项目发现 Hub.Close() 与 Unregister 并发 double-close channel 导致 panic。

- **禁止**在多个 goroutine 中对同一 channel 执行 `close()`。channel 的关闭权必须归属唯一的 owner goroutine，或使用 `sync.Once` 保护
- **禁止**在 graceful shutdown 中直接遍历清理资源而不先停止产生新事件的 goroutine。正确顺序：先停止事件循环 → 等待其退出 → 再统一清理资源

## 禁止查询过滤在应用层完成

来源：审查 Mailbox 项目发现 `since` 时间过滤在 Go 层做内存过滤，全量拉取后再筛选。

- **禁止**从 MongoDB 拉取全量数据后在 Go 代码中做过滤。所有过滤条件（时间范围、状态、名称等）必须下推到 MongoDB 查询的 `filter` 参数中
- **禁止**添加新的查询过滤维度时不同步创建 MongoDB 索引。无索引的过滤条件会导致全表扫描，随数据增长性能恶化

## 禁止前端过滤修改源数据

来源：审查 Mailbox 项目发现前端 `applyClientFilter` 直接覆盖原始数组，切换过滤器后数据永久丢失。

- **禁止**在 Vue 组件中对 `ref` / `reactive` 中的列表数据直接执行 `.filter()` / `.splice()` 等变异操作用于 UI 过滤。过滤/排序必须通过 `computed` 派生出新数组，原始数据源保持不变
- **禁止**WebSocket 或轮询收到的实时数据经过过滤后才存入状态。实时数据必须先无条件存入原始数据源，再由 computed 负责展示过滤

## 禁止前端放行无效输入

来源：UX 审查发现 ConditionEditor 的 BB Key 是手动输入框，策划打错后提交，后端返回校验错误但信息含技术术语看不懂。

- **禁止**在前端用自由文本输入枚举类值。BB Key、操作符、节点类型、result 等有限集合的字段，必须使用下拉选择器（`el-select`），不允许手动输入。同一数据源（如 BB Key 白名单）在所有使用处必须保持一致——`ConditionEditor` 和 `BtNodeEditor` 必须用相同的下拉列表
- **禁止**名称字段不做格式校验。所有 `name` 字段必须限制格式（小写字母开头 + 小写字母/数字/下划线，BT 额外允许 `/`），在前端输入时即时校验，不能等到后端返回 422 才提示
- **禁止**新建时不检查名称重复。前端必须在 name 输入框 blur 时调用 list API 检查是否已存在，给出明确的中文提示（如 `事件名称 "xxx" 已存在，请换一个`）。后端 MongoDB unique index 是兜底，不是唯一防线
- **禁止**NPC 表单保存时不检查行为树绑定完整性。如果状态机有 3 个状态但只绑定了 2 棵行为树，前端必须拦截并列出未绑定的状态名

## 禁止 URL 路径中的特殊字符未编码

来源：BT 行为树名称使用 `civilian/idle` 格式（含 `/`），导致 Vue Router 匹配失败、API 路径被错误分割、后端 `pathName` 只取到最后一段 `idle`。
二次来源：前端 `encodeURIComponent` 修复后，Go `http.ServeMux` 又自动把 `%2F` 解码回 `/` 写入 `r.URL.Path`，导致 `strings.Split(path, "/")` 仍然分割错误，编辑按钮报"记录不存在"。

- **禁止**将含有 URL 特殊字符（`/`、`?`、`#`、`%` 等）的 name 直接拼接到 URL 路径中。API 调用层必须使用 `encodeURIComponent()` 编码，后端必须 URL 解码（`url.PathUnescape`）
- **禁止**Vue Router 的动态路由段（`:name`）不处理含 `/` 的参数。如果参数可能含 `/`，路由必须使用 catch-all 语法（如 `:name(.*)`）
- **禁止**后端用 `strings.Split(r.URL.Path, "/")` 取最后一段作为含 `/` 的资源 name。Go `http.ServeMux` 会自动将 `%2F` 解码为 `/` 写入 `r.URL.Path`，导致分割结果错误。必须使用 `r.URL.RawPath`（保留编码）并按固定前缀截取，再 `url.PathUnescape` 解码。参见 `pathNameAfterPrefix` 实现
- **禁止**只修前端不验后端就认为 URL 编码问题已修复。含 `/` 的 name 涉及三层（前端路由 → 前端 API → 后端路径解析），必须端到端验证：在页面上点击编辑按钮，确认能正确加载数据

## 禁止前端表单对非技术用户不友好

来源：以策划/运营视角审查前端页面，发现多处"看不懂"——技术标签、空列表无引导、删除确认过于简略。

- **禁止**表单中只显示技术英文标签而无中文说明。所有表单字段下方必须有灰色提示文字，用策划能理解的语言解释字段含义和影响
- **禁止**下拉框依赖项为空时无引导。NPC 表单中 FSM/BT 下拉列表为空时，必须显示警告 + 跳转链接（如"暂无可用的状态机，请先创建状态机"）
- **禁止**列表页空数据时只显示空白表格。必须使用 `el-empty` 组件 + 引导按钮（如"创建第一个事件"）
- **禁止**删除确认弹窗只写"确认删除？"。必须明确说明删除对象名称和影响（如 `确认删除状态机「civilian」？使用此状态机的 NPC 将受影响。`）

## 禁止收到协作方请求后不回复就动手

来源：游戏服务端提出架构变更请求，ADMIN 侧直接开始写代码而未先回复确认，导致对方不知道进度和方案是否被接受。

- **禁止**收到姐妹项目（游戏服务端/Unity 客户端）的需求、方案、联调请求后，不先回复就开始编码。必须先回复对方：确认收到、表明态度（可行/需调整/有疑问）、说明大致计划
- **禁止**回复后直接跳到写代码。必须先同步文档（red-lines、dev-rules、spec 等），确保规范反映最新架构决策，再进入实现阶段
- **禁止**口头答应但不落文档。协作方提出的架构变更一旦确认，必须在相关文档（CLAUDE.md、spec、red-lines）中体现，避免后续开发遗忘上下文

## 禁止过度设计

- **禁止**实现用户认证/权限系统。毕设阶段所有用户等权
- **禁止**实现配置版本控制/回滚。JSON 文件已有 Git 版本控制
- **禁止**实现实时协作编辑。单人编辑足够
- **禁止**实现复杂的工作流审批。保存即生效（到 MongoDB）
