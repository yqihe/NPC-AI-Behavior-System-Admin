# 后端工程原则：从「写对」到「能删」

## 这份文档的定位

这不是原则清单，是**决策框架**。

- 清单让你「检查做没做到」，框架让你「判断该不该做、该做到什么程度」。
- 使用方式：遇到技术决策时，先定位它属于哪一层，再用那一层的判断标准做判断。
- 每一层都配了**本质判断标准**、**反例症状**、**落地要点**——不是拿来死记硬背，是拿来自己做推理。

## 核心命题

> **软件腐烂 = 隐式积累速度 > 显式化速度。**

后文所有原则都是这条命题在不同层面的具体形态。它们看起来各自独立，本质都在做同一件事——**把隐式的东西变成显式的**。

## 第一诊断问题

遇到任何技术决策时，先问自己一句：

> **「这件事显式化了吗？」**

- 隐式的 → 是**债**，它会在某个凌晨变成故障
- 显式的 → 是**资产**，它会降低未来的认知负担

把这个问题作为反射，其他原则会自然跟上。

---

## 使用前提：技术原则不在真空里运作

本文档描述的七层原则，**假设组织已具备下列三个前提**。否则再漂亮的技术原则也是悬空的。

### 前提一：具名责任

- 每个字段、API、SLO、架构决策都要有一个**具名的**主人
- **康威定律推论**：系统的「可负责性」反映组织的「可负责性设计」
- 无主人的字段 = 组织层面的债，伪装成技术债。留着的成本分摊给所有人，删的成本谁都不愿承担

### 前提二：演进的完整性

- 迁移要么做完，要么别开始——**半迁移会自我强化**：旧模式因存量多反而显得"更正统"，新人合理推断后继续沿用旧模式，形成正反馈循环
- ADR（架构决策记录）必须绑定流程节点（PR 模板、onboarding、架构评审），否则会腐烂成「谁都知道存在但没人读」的坟墓

### 前提三：显式化投入

- 显式化是**逆自然**的过程，没有主动投入，系统自发走向隐式堆积
- 一个团队真正的核心竞争力不是懂多少技术，是**能多快把知识从个人脑袋里迁到系统结构里**
- **写作能力 = 工程能力**：命名、注释、文档、ADR、接口设计全是写作的具体形态。写得差 = 显式化慢 = 腐烂快

## 七层递进

按「读者脑袋里的距离」从近到远组织——从「单行代码是否清晰」到「三年后这段代码能不能被安全删除」。每一层都以前一层为前提。

### 第一层：看得懂 — 读者能建模

**判断标准**：
> 「读完一个模块后，对其他模块能否形成合理的预期？」

**反例症状**：
- 开盲盒式代码——每打开一个文件都要重新学习
- `getUserInfo` / `get_order_detail` 混用
- A 接口错误格式 `{code, msg}`，B 接口 `{status, message}`
- 有的模块 Handler 里直接写 SQL，有的规规矩矩走 Service

**落地要点**：
- 命名风格（变量/函数/类/DB 字段/API 路径）全项目统一
- 分层结构（Controller → Service → Repository）全模块统一，不允许局部反常
- 错误处理格式、日志级别、注释风格都要有明文约定
- 本质：**可预测性**——让读者的大脑可以外推

### 第二层：可外推 — 调用者不读实现能预测行为

**判断标准**：
> 「调用者不读实现，能否正确预测返回值、字段语义、副作用？」

**反例症状**：
- 同类字段在不同模块叫不同名字（`created_at` vs `create_time`）
- 分页响应结构不一致（`{items, total}` vs `{list, count}`）
- 枚举值一处用字符串一处用整数
- 函数命名和内部行为不一致（`getXxx` 里藏着修改）

**落地要点**：
- **数据契约细化到字段级**：字段名、类型、可空性、枚举值全项目统一
- **抽象层级一致（SLAP - Single Level of Abstraction Principle）**：一个函数内部只讲一层的语言，不在业务语义里混 SQL 细节，不在业务流程里混错误包装
  - 检验方法：「把函数体朗读出来，能不能读成一个连贯的故事？」每隔一行切换语气 = 违反 SLAP
- 本质：**契约诚实**——签名不撒谎

### 第三层：可推理 — 不跑代码也能断言性质

**判断标准**：
> 「只靠函数签名和代码结构，能否推断出它的行为？」

**元原则**：
> **签名是契约，契约不能撒谎。**
> 签名说的是「我需要什么、给你什么」——任何不经签名的输入/输出都是隐式通道。

**反例症状**：
- 从 `ctx.Value` 里掏业务字段（让类型系统失明，工具链不透明）
- 函数内部调 `time.Now()` / `rand.Intn()` / `os.Getenv()`（隐藏入参）
- 函数里 `go func()` 起飞（调用者看不到还有异步任务在跑）
- 同一类 API 超时一会儿重试一会儿 fail-fast（失败语义不一致）
- 查询函数里偷偷更新 `last_active_time`（副作用不透明）

**落地要点**：

**副作用显式化**（输出侧的诚实）：
- 核心逻辑保持纯函数，副作用集中在最外层一两层（命令层）——**Functional Core, Imperative Shell**
- 查询和命令分离（CQRS 的核心动机之一）
- 不把副作用藏在名字看不出来的函数里
- Go 里尤其注意：`go func()`、defer 链、channel 起飞都是隐式副作用

**隐式依赖显式化**（输入侧的诚实）：
- 时间通过 `Clock` 接口传入，不直接 `time.Now()`
- 随机源通过 `rand.Source` 传入
- 配置在调用前读取、作为参数传入，不在函数内部 `viper.Get(...)`
- `ctx.Value` 只用于日志追踪类非业务字段，业务数据**不从 ctx 里掏**
- **判断标准**：一个值要不要显式传参，看它**是否需要被工具链追踪**。需要 → 必须走签名

**失败语义一致**：
- 按「失败类别」定义统一响应策略：下游超时→重试/降级/fail-fast？ 缓存 miss→回源/返空/报错？ 外部不可用→熔断/堆队列？
- **熔断的本质不是"保护下游"，是主动定义失败的传播边界**——这里断，不往上蔓延
- 系统的「可用性画像」必须被显式讨论，不是各模块直觉叠加。支付 vs 推荐 vs 核销对「可用」的定义完全不同

**暗通道警报**：
所有绕过类型系统的设计都是暗通道，不止 `ctx.Value`：
- `map[string]interface{}` / `any` 配运行时断言
- 反射驱动的"通用工具"
- 字符串拼的"事件名"
- JSON 作为内部模块间交换格式

暗通道的共同特征：**编译器看不见、IDE 跳不到、重构没有保护**。一个系统的可维护性，很大程度上取决于它有多少暗物质。

### 第四层：可保证 — 性质被结构锁死

**判断标准**：
> 「非法状态在类型系统中能被表达吗？能表达 = 结构不够强。」

**元原则**：
> **让代码从「靠自觉」升级到「靠结构」。**

**反例症状**：
- 满屏 `if x == nil / if x < 0 / if !x.IsValid()`——**防御代码是不变式没锁住的症状**
- 防御代码会**自我繁殖**：调用方不信任被调用方 → 被调用方也对自己的输入防御 → 整个系统变成互不信任的网络
- 防御代码会**吃掉 panic**：nil check 加默认值继续执行，异常降级成脏数据，在离根因很远处才爆
- `status == "paid"` 的检查散落在十几个 service 方法里，每个入口各自守

**落地要点**：

**不变式类型**（全都是显式化候选）：
- 业务不变式（订单金额 = 明细之和）
- 集合不变式（活跃用户数 ≤ 总用户数）
- 状态机不变式（Paid 之后不可能回 Pending）
- 守恒不变式（库存扣减总量 = 已售）
- ACID 不变式（事务原子性）

**五种维持手段**（按强度从高到低）：
1. **类型系统锁死**：`type Email string` + 强制构造函数；**分态 struct** 让非法流程在编译期就不存在
   ```go
   type UnvalidatedOrder struct { ... }
   type ValidatedOrder   struct { ... }  // 不同类型，不是 bool 字段
   type PaidOrder        struct { ... }

   func (u *UnvalidatedOrder) Validate() (*ValidatedOrder, error)
   func (v *ValidatedOrder)   Charge()   (*PaidOrder, error)
   ```
2. **构造函数强制**：禁止 `&Order{}` 裸构造，只允许 `NewOrder(...)`，校验在出生时完成
3. **状态机集中化**：把转移规则显式写成数据结构
   ```go
   var validTransitions = map[Status][]Status{
       Pending: {Paid, Cancelled},
       Paid:    {Refunded},
       // Paid → Pending 写都写不进来
   }
   ```
   额外红利：可视化免费（可生成 Graphviz）、测试自动生成、审计完备
4. **Property-based testing**：声明「对任意输入这个性质必须成立」，让测试自动找反例。比 example-based 高一个维度
5. **数据库约束**：CHECK / UNIQUE / FOREIGN KEY——最后的防线，不依赖应用层自觉

**核心心智切换**：
从「**我在哪里加校验**」切换到「**我怎么建模让校验变得不必要**」。这是过程式心智到声明式心智的迁移，技巧易传，习惯难养。

### 第五层：可观测 — 异常能被运行时感知

**判断标准**：
> 「三年后凌晨三点被这个模块的告警叫醒，我需要什么信息才能 10 分钟内定位问题？现在埋好了吗？」

**元原则**：
> **告警的本质不是「指标超阈值」，是「断言和观察之间出现了分歧」。**
> 不变式（第四层）= 设计时对「世界应有的样子」的断言。
> 可观测性 = 运行时对「世界实际的样子」的观察。
> bug = 两者的差异。

**反例症状**：
- 非结构化日志（ELK 里基本是废的——查不到、聚合不了、告警写不出）
- 跨服务/跨异步任务时 trace_id 断掉
- 告警基于 CPU / 内存这类症状，而不是业务断言偏离
- 写过 ADR 但没人读，产生「我们有决策记录」的错觉

**落地要点**：

**四个维度（互补关系，不能互相替代）**：
- **Metrics**（聚合）：耗时、成功率、重试次数、队列深度
- **Tracing**（单次链路）：trace_id 一路传到底，包括异步任务
- **Logs**（结构化流）：必须带 schema，不是备忘录
- **Events / 审计日志**（离散、因果、长期存储）：metrics 聚合掉了个体，logs 没有因果——这一维单独建（event store / 审计表）

**SLO 驱动设计**：
- 把「系统应该是什么样」写成可验证的 SLO
- 运行时持续比对，仅在期望与现实背离时告警
- 优于「CPU > 80% 就响」这种盲目报警

**3 a.m. 自测法**：
- 写代码时换位到未来陷入故障的自己
- 需求是**那时的自己**需要什么信息，不是**现在的自己**想记什么日志

### 第六层：可负担 — 复杂度在预算内

**判断标准**：
> 「每加一层抽象/中间件/配置项，它解决的**具体**问题是什么？没有它会**多 painful**？如果答不上来，不加。」

**元原则**：
> **企业级 ≠ 所有模式都上。企业级 = 在预算内选最少的模式。**

**反例症状**：
- "企业级" 被理解成 "所有设计模式全开"
- 配置项堆成数百条，大多是"以防万一"
- 抽象层数过深，每层只解决假想问题
- 同一份数据穿了 5 个 DTO 来回转
- Feature flag 堆了 50 个，全是 `true`，没人敢删

**落地要点**：

**显式预算**（必须写下来的几个维度）：
- 抽象层数上限
- 外部依赖数量
- 模块间耦合深度
- 配置项数量
- Feature flag 数量（**每个 flag 出生就带死期**）

**每次加东西前三问**：
- 这一层解决了哪个**具体**问题？
- 没有它会**多 painful**？
- 当前有两个以上的使用方吗？没有 → 不抽。

**合题：预算本身也要被显式化。**

没有写下来的预算，就是每个人心里刻度盘不同的**审美争论**。「保持简单」是一句口号，不是一条原则。
这一层本身也要服从这一层——预算管理不能是拍脑袋。和「不变式锁在类型里」是同构动作：**把「靠自觉」升级成「靠结构」**，只不过这一次在团队层面。

### 第七层：可删除 — 代码能被安全撤回

**判断标准**：
> 「这段代码能被安全删除吗？删除要花多长时间确认影响范围？」

**元原则**：
> **团队敢不敢删代码，是比任何代码评审都诚实的系统健康度指标。**
> 代码评审可以走形式，覆盖率可以刷数字，架构图可以画得很漂亮——但「这个季度我们删了多少代码」是客观数字。

**关键反推**：

可删除性是前六层的**压力测试**——做到了，它几乎是副产品；做不到，前面一定有条没落地。

| 前层做到 | 带来的可删除性红利 |
|---|---|
| 命名/结构一致 | grep 能找到所有相关代码 |
| 契约诚实 | 依赖关系可追踪 |
| 副作用显式 | 知道删掉它会改变什么 |
| 不变式锁住 | 类型系统指出所有被打破的地方 |
| 可观测 | 灰度阶段能确认无影响 |
| 复杂度预算 | 没留"以防万一"的遗产 |

**三层删除（越往下越难）**：

1. **代码层**：死代码，静态分析能扫
2. **功能层**：废弃 API，走 deprecation 周期、对外沟通、观察调用归零
3. **数据层**：废弃字段/表——**极难**
   - 数据库 schema 变更成本远高于代码 → 形成「加字段安全、删字段危险」的**不对称变更成本** → 天然堆积
   - **解法不在技术层，在治理层**：每个字段有具名主人（参见使用前提一）
   - 无主人字段 = 无人认领的技术债，留着的成本分摊给所有人，删的成本谁都不愿承担

**代码半衰期**：

每段代码应对"自己能活多久"有诚实预期：
- 一次性迁移脚本 → 完成即删
- 过渡适配层 → 旧系统下线即删
- 核心领域模型 → 产品生命周期
- 底层工具 → 跨产品

大多数团队不区分 → 一律按"永远存在"来设计 → **一次性脚本被当成了永久基础设施，没人敢删，因为没人记得它为什么存在**。

**落地方式**（创建时就标注预期下线条件）：
```go
// 一次性数据迁移，2026-Q2 完成后删除
// feature flag: 2026-06-30 前必须完成清理
// LegacyXxx 前缀：直到旧系统下线
```

**对抗防御式编程的哲学**：

> **宁可在 root cause 处 panic，不要在远端拿到脏数据。**

防御代码看起来"健壮"，实际上是把**确定性故障**换成了**不确定性数据污染**——几乎总是亏的交易。Crash-only software 和 fail-fast 的价值就在这里。

## 终极收束

> **一致性的终极目标，是让系统从「靠纪律维持」升级到「靠结构保证」。纪律会疲劳，结构不会。**

**三层成熟度**：
- **能写对**（初级）：第 1-2 层
- **能改对**（高级）：第 3-5 层
- **敢删对**（系统/架构师）：第 6-7 层 + 三个使用前提

**毕设代码 vs 企业代码的根本差别**：
- 毕设代码被设计成「**完成**」——一个有始有终的作品，交付即结束
- 企业代码被设计成「**持续演化**」——没有终点，永远在被添加、修改、删除、替换

企业级工程真正要对抗的不是功能复杂度，是**时间**——时间里堆积的依赖、时间里变迁的需求、时间里离开的人。本文档所有原则，本质上都在回答同一个问题：

> **三年后、五年后、团队换了一轮之后，这段代码还能被理解、被修改、被删除吗？**

## 反诊断清单

遇到下列症状，对照回查：

| 症状 | 很可能是哪一层没做到 |
|---|---|
| 防御代码泛滥（nil check 满屏） | 第 4 层——不变式没锁住 |
| 排查问题要翻多个人的微信 | 使用前提一（具名责任）+ 前提三（显式化） |
| 没人敢删代码 | 前 6 层有欠账 |
| 告警基于 CPU/内存等症状 | 第 5 层——停留在表层观测 |
| 架构评审全靠「我觉得」 | 第 6 层——没有预算 |
| 字段没人认识是干什么的 | 使用前提一——具名责任缺失 |
| 函数签名干净但行为不可预测 | 第 3 层——隐式依赖 |
| 同一类失败各模块反应不同 | 第 3 层——失败语义不一致 |
| 新人入职 3 个月才敢改代码 | 使用前提三——显式化投入不足 |
| 半年前加的抽象层没人用 | 第 6 层——预算失控 |
| 删一个函数要花三天确认影响 | 前 6 层综合不足 |

---

*本文档的形成过程本身，就是一次把个人直觉显式化的尝试。如果未来某天它过时了，最好的对待方式不是绕过，是用同样的方式**显式地**更新它——命名新问题、找到新边界、把新的隐式变成新的显式。*
