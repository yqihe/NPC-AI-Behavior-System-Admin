# 生命周期功能测试指南 — 村庄守卫 NPC

## 背景故事

你是一名策划，正在为一款 RPG 游戏配置一个"村庄守卫（Guard）"NPC。
这个守卫会在村口**巡逻**，感知到敌人后进入**警戒**，确认威胁后**战斗**，威胁解除后**撤退**回到初始状态。

本指南将带你完整走过以下模块，逐一验证所有关键功能：

```
字段管理 → 模板管理 → 事件类型 Schema → 事件类型 → 状态字典 → 状态机
```

---

## 前置准备

1. 确认系统已启动（`docker compose up` 或本地启动）
2. 打开浏览器访问前端 `http://localhost:3000`
3. 确认后端接口正常（可访问 `http://localhost:9821/api/v1/fields/list`）

---

| 约束 → max | `100` |

## PHASE 6：状态机管理

> **目标**：为守卫创建完整 FSM，验证状态配置、转换条件编辑器（AND/OR 嵌套）、版本控制。

### 6.1 创建状态机

进入**状态机管理** → 点击**新建**：

| 项目 | 值 |
|------|-----|
| 标识名 | `guard_fsm` |
| 显示名 | 村庄守卫状态机 |
| 初始状态 | `idle` |

**添加状态**（按此顺序）：

```
idle
patrolling
alerting
in_combat
retreating
```

**添加转换规则**（共 6 条，Priority 越小优先级越高）：

---

#### 转换 1：待机 → 巡逻（无条件，游戏开始后自动进入巡逻）

| 项目 | 值 |
|------|-----|
| From | `idle` |
| To | `patrolling` |
| Priority | `10` |
| 条件 | **空条件**（始终成立） |

---

#### 转换 2：巡逻 → 警戒（发现可疑情况）

| 项目 | 值 |
|------|-----|
| From | `patrolling` |
| To | `alerting` |
| Priority | `5` |
| 条件类型 | **单条件（Leaf）** |
| Key | `guard_is_alerted` |
| Op | `==` |
| Value | `true` |

---

#### 转换 3：警戒 → 战斗（确认威胁，血量充足）

| 项目 | 值 |
|------|-----|
| From | `alerting` |
| To | `in_combat` |
| Priority | `3` |
| 条件类型 | **AND 组合** |
| 子条件 1 | `threat_level` `>=` `5` |
| 子条件 2 | `guard_hp` `>` `30.0` |

> 操作提示：点击"添加条件组 AND" → 在 AND 组内添加 2 条 Leaf 条件

---

#### 转换 4：警戒 → 巡逻（虚惊一场）

| 项目 | 值 |
|------|-----|
| From | `alerting` |
| To | `patrolling` |
| Priority | `8` |
| 条件类型 | **单条件（Leaf）** |
| Key | `guard_is_alerted` |
| Op | `==` |
| Value | `false` |

---

#### 转换 5：战斗 → 撤退（血量过低 OR 遇到 Boss）

| 项目 | 值 |
|------|-----|
| From | `in_combat` |
| To | `retreating` |
| Priority | `1` |
| 条件类型 | **OR 组合** |
| 子条件 1 | `guard_hp` `<` `20.0` |
| 子条件 2 | `is_boss` `==` `true` |

> 操作提示：点击"添加条件组 OR" → 在 OR 组内添加 2 条 Leaf 条件

---

#### 转换 6：撤退 → 待机（返回初始位置）

| 项目 | 值 |
|------|-----|
| From | `retreating` |
| To | `idle` |
| Priority | `10` |
| 条件 | **空条件**（始终成立） |

---

保存 FSM。
